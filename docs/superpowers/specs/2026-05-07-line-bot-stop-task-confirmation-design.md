# 規格說明：Line Bot 停止任務二次確認機制

## 1. 背景與目的
目前使用者在點擊「🛑 停止任務」按鈕時，系統會立即執行終止操作。這在行動裝置上容易因誤觸導致正在進行的監控任務無意中斷。本規格旨在透過 LINE 的 Quick Reply (快捷按鈕) 實作二次確認機制，提昇系統的穩定性與操作安全性。

## 2. 使用者流程 (User Flow)
1. **觸發**：使用者輸入「查詢」，系統回傳任務輪播卡片。
2. **點擊**：使用者點擊某個任務下方的「🛑 停止任務」。
3. **確認請求**：
   - 系統不立即停止任務。
   - 系統發送訊息：「⚠️ 您確定要停止此任務嗎？\n[路線資訊]」
   - 訊息下方顯示 Quick Reply 快捷按鈕：
     - `✅ 是，確定停止` (文字指令: `確認取消:是:{index}`)
     - `❌ 否，繼續監控` (文字指令: `確認取消:否`)
4. **最終執行**：
   - 若點擊「✅ 是」，系統執行 `monitor.stop()`，移除任務，並回覆成功訊息。
   - 若點擊「❌ 否」，系統回覆「操作已取消，任務繼續執行」，且不更動任務狀態。

## 3. 技術組件
### 3.1 輔助函數
- **`create_confirm_cancel_quick_reply(idx, info_text)`**:
  - 參數：任務索引 `idx`, 任務簡短描述 `info_text`。
  - 回傳：`QuickReply` 物件。

### 3.2 訊息處理器 (handle_message)
- **更新 `取消任務:{idx}`**：
  - 改為查詢任務資訊。
  - 發送帶有二次確認 Quick Reply 的 Flex Message 或 Text Message。
- **新增 `確認取消:是:{idx}`**：
  - 驗證 `idx` 是否仍有效。
  - 執行停止與移除邏輯。
- **新增 `確認取消:否`**：
  - 直接回覆取消提示。

## 4. 錯誤處理
- 若在確認過程中任務已由系統自動完成（搶到票），則點擊「確認」時應提示「該任務已結束」。

## 5. 驗證標準
- [ ] 點擊「停止任務」後不會立即停止。
- [ ] 底部出現兩個對比明顯的確認按鈕。
- [ ] 點擊「否」後任務清單依然存在。
- [ ] 點擊「是」後任務確實停止且清單移除該項。
